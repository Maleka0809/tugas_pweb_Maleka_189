Testing dan Debugging

1. Analisis Kode Program

Kode program terdiri dari tiga kelas utama yang saling berinteraksi untuk mensimulasikan sistem komentar pada sebuah produk penjualan sederhana.

A. Kelas Comment

Kelas ini berfungsi sebagai struktur data untuk menyimpan detail sebuah ulasan.

  • Atribut: Menyimpan nama penulis (author), isi ulasan (text), dan nilai rating (rating).
  • Fungsi: Menyediakan getter untuk mengakses data dan metode printDetails() untuk mencetak ulasan ke layar.

B. Kelas SalesItem

Ini adalah kelas utama yang merepresentasikan produk.

  • Struktur Data: Menggunakan ArrayList<Comment> untuk menampung jumlah komentar yang dinamis (bisa bertambah terus).
  • Logika Validasi: Pada method addComment, terdapat logika validasi (Boundaries Check). Komentar hanya akan disimpan jika rating berada dalam rentang 1 hingga 5. Jika di luar itu, metode mengembalikan false.

C. Kelas SalesItemTest (Unit Testing)

Kelas ini menggunakan kerangka kerja JUnit yang terintegrasi di BlueJ. Tujuannya adalah untuk mengotomatisasi pengujian logika tanpa harus menjalankannya secara manual berulang kali.

2. Laporan Pengujian (Testing)

Pengujian dilakukan dengan dua metode: Automated Unit Testing dan Manual Testing.

Skenario Pengujian

Terdapat dua skenario utama yang diuji untuk memastikan validitas data:

  • Positive Case (Data Valid): Memasukkan rating 5.
    Ekspektasi: Metode addComment mengembalikan true dan jumlah komentar bertambah 1.
  • Negative Case (Data Invalid): Memasukkan rating 6 (di luar batas).
    Ekspektasi: Metode addComment mengembalikan false dan jumlah komentar tetap 0 (data ditolak).

Hasil Pengujian

  • Hasil Unit Test (JUnit):
    • Saat SalesItemTest dijalankan, BlueJ menampilkan Green Bar (Bar Hijau).
    • Ini menandakan bahwa testAddComment (kasus positif) dan testInvalidRating (kasus negatif) telah lolos asersi (assertEquals dan assertTrue/False sesuai).
  • Hasil Manual Test:
    • Objek SalesItem dibuat di Object Bench.
    • Pemanggilan addComment("User", "Test", 6) menghasilkan return false.
    • Pemanggilan showInfo() menunjukkan "Jumlah Komentar: 0".
    • Kesimpulan: Logika validasi if-else berfungsi dengan benar.

3. Proses Debugging

Proses debugging disiapkan untuk menangani potensi kesalahan logika (logic error) di mana program berjalan namun hasilnya salah.

Metodologi Debugging

Alat yang digunakan adalah BlueJ Debugger. Berikut adalah langkah simulasi penelusuran alur program:

  1. Penempatan Breakpoint:
    Sebuah Stop Sign (Breakpoint) ditempatkan pada baris if(rating < 1 || rating > 5) di dalam kelas SalesItem. Tujuannya untuk menghentikan program tepat sebelum validasi rating dieksekusi.
  2. Inspeksi Variabel:
    Saat metode dipanggil dengan input rating 6, program berhenti (paused). Panel debugger menunjukkan status variabel saat ini:
    • rating: 6
    • author: "Andi"
  3. Stepping (Penelusuran):
    Menggunakan tombol Step, alur program terlihat masuk ke dalam blok if. Ini mengonfirmasi bahwa kondisi rating > 5 terpenuhi (bernilai true), sehingga program mengeksekusi perintah return false.
  4. Verifikasi:
    Proses ini memastikan bahwa blok kode pembuatan objek new Comment(...) dilewati/di-skip ketika rating salah, sehingga mencegah data sampah masuk ke dalam sistem.

4. Kesimpulan

Berdasarkan hasil testing dan debugging:

  • Kode SalesItem telah memiliki proteksi data yang baik terhadap input rating yang tidak valid.
  • ArrayList berfungsi dengan baik dalam menyimpan objek Comment.
  • Unit Test mempermudah verifikasi fungsi setiap kali ada perubahan kode di masa depan (Regression Testing).
